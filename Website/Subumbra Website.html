How it works

One bootstrap. Zero plaintext — ever.

A one-shot setup encrypts your keys and deploys the decryption authority to Cloudflare. After that, your servers only ever hold useless ciphertext.

01 / Bootstrap
Keys enter RAM only
Run the interactive wizard. Your API keys exist only in process memory — never written to disk at any point.
docker compose --profile bootstrap
run --rm -it bootstrap
02 / Encrypt
RSA-4096 + AES-256-GCM
Each key gets a unique DEK wrapped by an RSA-4096 public key. AAD binding ties each ciphertext to its key ID — transplant attacks fail.
subumbra:v2:<key_id>
AES-256-GCM · OAEP-SHA256
03 / Split
Private key goes to Cloudflare
The RSA private key is pushed directly to Cloudflare Secrets and deleted from memory. Your VPS holds ciphertext only.
WORKER_PRIVATE_KEY → CF Secrets
keys.json → your volume
04 / Proxy
Apps use a plain key ID
Apps point at subumbra-proxy with a plain key_id. The Durable Object decrypts for ~100ms, calls the provider, and the key is gone.
api_base: http://subumbra-proxy
api_key: anthropic_prod

Features

Everything a team needs to manage secrets safely.

⚡
Transparent proxy
Drop-in sidecar at :8090/t. No app code changes — swap the base URL and use a key ID instead of a real key.
🔄
Offline key rotation
Rotate individual keys using the public key on disk. No Cloudflare interaction, no Worker redeploy, no service restart. Zero downtime.
🧩
Live provider registry
Provider allowlists live in Cloudflare KV. Add or update providers without redeploying the Worker. Custom providers fully supported.
📊
Management dashboard
Health status, per-key request counts, last access times, and a live structured audit log — without ever displaying key values.
🔒
Secure browser key entry
Add or rotate keys via the browser with ephemeral RSA session keypairs and SubtleCrypto paste interception. Plaintext never touches the DOM.
🧱
Multi-adapter support
Named adapters with scoped key access, independent tokens, and configurable expiry. OpenWebUI, AnythingLLM, n8n, and custom apps all work.

Security model

Designed so a partial compromise yields nothing useful.

The split-decrypt architecture means an attacker needs to compromise both your infrastructure and Cloudflare simultaneously to recover a key.

  • ✓
    RSA-4096 + AES-256-GCM envelope encryption
    Per-record DEKs wrapped by RSA-4096. AAD binding prevents ciphertext transplant between records.
  • ✓
    Private key lives only in Cloudflare Secrets
    Your VPS holds encrypted blobs — useless without the private key. Neither side alone is enough.
  • ✓
    Durable Object: one-time decrypt, ~100ms
    Decryption happens in an isolated CF Durable Object. The key is used to call the provider and immediately destroyed. Every call is logged.
  • ✓
    HMAC-signed, replay-protected fetches
    Every ciphertext fetch requires a per-request HMAC signature with a single-use nonce and timestamp tolerance window.
  • ✓
    Fail-closed provider validation
    The Worker validates target hostnames and provider identity against a live Cloudflare KV registry before decrypting. Mismatches hard-fail.
  • ✓
    Durable structured audit log
    Every key fetch, denial, and rotation is written to a local SQLite audit database with verdict, reason code, adapter ID, and remote IP.
