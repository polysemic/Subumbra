# Subumbra 1.1.0-alpha Release Notes

*Release date: 2026-05-19*

This release advances Subumbra from `v1.0.0-alpha` to `1.1.0-alpha` with three
main themes:

- Cloudflare runtime credential UX improvements
- optional Cloudflare auto-provisioning and teardown
- source-trust and deployment-integrity verification before bootstrap secrets

Compare:

- `v1.0.0-alpha...1.1.0-alpha`

---

## Highlights

### Cloudflare setup is much more complete

Subumbra can now support three practical Cloudflare paths:

- minimal Worker + KV only
- bring-your-own Tunnel / Access runtime credentials
- bootstrap-managed Cloudflare lifecycle with optional auto-provisioning

This makes first-time setup and day-2 operations much clearer for operators who
want either a minimal setup or a more fully managed Cloudflare path.

### Bootstrap can now auto-provision Cloudflare resources

Bootstrap can now optionally create and later tear down:

- Cloudflare Tunnel
- DNS CNAME records
- Cloudflare Access app
- service-auth policy
- service token

When used, generated runtime credentials are written into `.env`, and
non-secret Cloudflare resource IDs are tracked in `data/cf-resources.json`.

### New source-trust verifier before secrets are entered

This release adds `scripts/subumbra-verify`, a read-only integrity checker that
helps operators inspect the local checkout before entering Cloudflare or
provider credentials. Bootstrap now runs a preflight verifier automatically
before reading `.env.bootstrap` or prompting for secrets.

## What Changed Since v1.0.0-alpha

### Added

- Optional bootstrap ingestion of BYOC Cloudflare runtime credentials:
  `TUNNEL_TOKEN`, `CF_ACCESS_CLIENT_ID`, and `CF_ACCESS_CLIENT_SECRET`
- Day-2 runtime credential update commands:
  `./bootstrap.sh --update-tunnel` and `./bootstrap.sh --update-access`
- Optional Cloudflare auto-provisioning during bootstrap for Tunnel, DNS, and
  Access resources
- `./bootstrap.sh --nuke-cloudflare` for teardown of tracked Cloudflare-managed
  resources
- `data/cf-resources.json` to track non-secret Cloudflare resource IDs
- `scripts/subumbra-verify` for local source-trust checks
- stricter bootstrap preflight before secrets are read
- published release-signing public key path at
  `docs/release-signing-key.pub`
- public security scan/reporting layout under `security/reports/`
- VPS-oriented public security-suite runner and supporting scan-tool scripts

### Changed

- Interactive bootstrap now offers optional Tunnel / Access prompts with reuse
  of existing `.env` values
- Cloudflare planning is no longer treated as an active product theme; it is
  now debug- or request-driven only
- install, developer, and security docs now describe normal verification flow,
  stricter signed-tag flow, and the release-signing trust path
- roadmap and project-memory docs now reflect the completed Cloudflare arc and
  `1.1.0-alpha` baseline

### Security and integrity

- `bootstrap.sh` now runs `./scripts/subumbra-verify --preflight` before
  `.env.bootstrap` is read or interactive secret prompts begin
- `scripts/subumbra-verify` checks:
  - Git worktree state
  - drift in selected sensitive files
  - basic local manifest / state shape
  - obvious bootstrap-secret residue
  - optional read-only Worker drift via `subumbra-verify-deploy`
- strict signed-release enforcement is supported through
  `SUBUMBRA_REQUIRE_SIGNED_TAG=1`
- release-signing documentation now distinguishes template-catalog signing from
  Git release-tag signing

## Upgrade Notes

- Existing operators upgrading from `v1.0.0-alpha` should review:
  - [docs/subumbra-install.md](../subumbra-install.md)
  - [docs/cloudflare-setup.md](../cloudflare-setup.md)
  - [docs/cloudflare-tunnel-access.md](../cloudflare-tunnel-access.md)
  - [docs/operator-guide.md](../operator-guide.md)
- If you use Cloudflare runtime credentials, `r72` adds first-class update
  verbs rather than requiring full re-bootstrap just to rotate those values.
- If you want bootstrap-managed Cloudflare lifecycle, review the `r73` flow
  before using auto-provisioning in a live environment.
- If you want a stricter source-trust workflow, review `scripts/subumbra-verify`
  and signed-release verification before using
  `SUBUMBRA_REQUIRE_SIGNED_TAG=1`.

## Known Notes

- This is still an **alpha** release.
- Current strict signed-tag verification is now supported, but it depends on
  using signed annotated release tags and a trusted public signing key.
- The `ws` advisory currently visible through GitHub security tooling is still
  upstream in the Wrangler/Miniflare dependency chain as of this release.
- Streaming response scanning remains deferred; `response.deny_patterns` still
  applies only to buffered response types.
